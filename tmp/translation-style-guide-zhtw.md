# translation-style-guide-zhtw

## 目的

這份文件定義 repo 文件翻譯成繁體中文時的寫作規範，確保技術正確、可操作、可維護。

適用範圍：

- `README.md`
- `guides/` 中仍保留為文字文件的內容
- `workshop/docs/`
- `workshop/mkdocs.yml` 中會直接顯示給使用者的文案
- `infra/vscode_web/` 的使用者導向說明文件

## 核心原則

1. 以 `workshop/docs/` 為唯一主要內容來源。
2. 先保技術正確，再追求語句流暢。
3. 不為了中文通順而改動命令、路徑、參數與程式碼。
4. 保持與英文原文相同的操作順序與資訊層級。
5. 標題、導覽與章節命名需彼此一致。

## 風格原則

### 語氣

- 使用專業、清楚、直接的繁體中文。
- 避免口語化、行銷化、過度熱情的措辭。
- 優先寫成「可執行」的說明，不寫模糊描述。

### 人稱

- 優先使用中性描述，例如：
  - 「請執行以下指令」
  - 「確認你已具備下列權限」
- 避免大量使用「你會看到」「你將會」這類較口語寫法，除非是引導步驟。

### 句型

- 步驟用短句。
- 複雜概念先給一句總結，再補細節。
- 表格欄位名稱保持短而清楚。

## 不可翻譯項目

以下內容必須保持原文：

1. CLI 指令
2. code block 全文
3. 環境變數名稱
4. 檔名與路徑
5. JSON/YAML key
6. API 與 SDK 符號名稱
7. URL 與連結目標

示例：

```bash
azd up
python scripts/00_build_solution.py --from 02
```

```env
FABRIC_WORKSPACE_ID=your-workspace-id
```

## 可翻譯項目

以下內容應翻譯：

1. 章節標題
2. 一般敘述句
3. 表格中的說明欄位
4. admonition 標題與內容
5. 導覽列文字

## 專有名詞規則

依 [translation-glossary-zhtw.md](translation-glossary-zhtw.md) 為準。

補充規則：

1. 產品正式名稱優先保留英文。
2. 技術概念首次出現時，可寫成「中文（英文）」。
3. 同一篇內不要同時混用兩種譯法。

## Markdown 保留規則

翻譯時必須保留：

1. 標題層級
2. 清單層級
3. 表格欄數與格式
4. 連結目標
5. admonition 語法
6. tabbed block 與 code fence 語法

不得因翻譯而破壞：

- `!!! note`
- `=== "Windows"`
- 表格分隔線
- 相對路徑連結

## 導覽與標題規則

### MkDocs 導覽

- 導覽名稱可翻譯，但需與頁面標題對齊。
- 若頁面標題保留英文，導覽應維持一致。

### 檔名

- 檔名不翻。
- 只翻頁面內容與導覽顯示名稱。

## 指令與步驟規則

### 指令前導句

建議寫法：

- 「執行以下指令以部署 Azure 資源：」
- 「更新 `.env` 後，再執行下列命令：」

避免寫法：

- 「把這串 code 打進去」
- 「隨便執行下面指令」

### 步驟編號

- 英文若是有順序的 numbered list，繁中也保留 numbered list。
- 不要把有順序的操作改成無序 bullet。

## 中英混排規則

1. 產品名保留英文時，周邊敘述用中文。
2. 程式碼與中文之間保留自然空白。
3. 中文標點為主，但 code、路徑、環境變數周邊不要插入不必要全形符號。

示例：

- 正確：請先執行 `azd auth login`。
- 正確：確認 `FABRIC_WORKSPACE_ID` 已設定。
- 避免：請先執行「`azd auth login`」。

## 連結處理規則

1. 相對連結目標不可改壞。
2. 若只翻連結文字，不要改括號中的路徑。
3. 錨點若來自標題自動生成，翻譯後需重新驗證是否仍可正確跳轉。

## 表格處理規則

1. 保持相同欄位數量。
2. 技術欄位值若是產品名、指令、檔名則保留英文。
3. 描述欄翻譯後不要讓表格結構失衡。

## Reviewer 判斷優先順序

review 時請依下列順序判斷：

1. 技術內容是否仍正確
2. 指令與路徑是否完全未破壞
3. 術語是否符合 glossary
4. 導覽、標題、連結是否一致
5. 中文是否自然好讀

## 常見錯誤

1. 把 `workspace` 在同一篇內翻成 工作區 和 工作空間
2. 把 `Agent`、`agent`、`代理程式` 混用且沒有規則
3. 把 code block 裡的註解也一併翻壞縮排或語法
4. 翻譯表格時改動欄位結構
5. 翻譯 `mkdocs.yml` 時只改導覽，不改對應頁面標題

## 完成定義

一篇文件完成翻譯，至少要同時滿足：

1. 術語符合 glossary
2. 指令、路徑、環境變數保持正確
3. Markdown 格式未破壞
4. 與相鄰頁面的導覽名稱一致
5. reviewer checklist 可全部勾選
