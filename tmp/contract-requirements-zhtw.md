# 合約需求整理

本文根據 `ref/requests.txt` 與附件內容整理，目的是把目前新增的兩組需求轉成可規劃、可拆解、可實作的 POC 輸入。

## 一、需求分成兩條主線

### 1. AI 合約審閱

- 情境一：全文審閱，尚待研究
- 情境二：關鍵字審閱，已上線，可作為現況參考

### 2. 自動化產生合約

- 以合約生成為核心
- 後續串接聲請單與審核流程
- 目前已有使用者需求文件、POC 合約範本與一版架構圖

## 二、AI 合約審閱需求重點

### 1. 全文審閱目標流程

使用者先在內部表單系統上傳前版本合約與此次合約，系統以 Python 產出差異比較 HTML，之後 AI 同時讀取前版本、此次版本、差異比較檔與規則檔，再依規則生成審閱意見。

可整理成四個步驟：

1. 上傳前版與新版合約
2. 產生差異比較 HTML
3. 讀取規則檔與合約內容
4. 生成結構化審閱意見

### 2. 關鍵字審閱現況流程

已上線版本只要求 AI 讀取差異比較檔，再依規則檔中的關鍵字做審閱，不需要完整理解兩份合約全文。

這代表現況系統偏向：

- 規則驅動
- 關鍵字比對
- 差異片段導向

而待研究的全文審閱則會走向：

- 文件理解
- 差異與上下文並讀
- 規則推理
- 生成更完整的法務意見

### 3. 規則檔的實際結構

附件四規則檔已不是單純的關鍵字表，而是接近一份規則知識庫。主要欄位包含：

- 規則編號
- 規則名稱
- 適用合約類型
- 參照條文位置
- 關鍵字
- 需 AI 判斷內容
- 可改 / 不可改
- 若比對結果不同時的提示語

目前已整理出的高價值規則類型包括：

- 合約標題
- 甲乙方資訊
- 引言
- 個案執行細節
- 付款條件
- 保密條款
- 智慧財產權
- 遲延與違約責任
- 合約有效期間
- 廉潔條款
- 職業安全衛生條款
- 資訊安全條款
- 委外廠商個資條款

這些規則同時標示哪些條款可調整、哪些原則上不可調整，代表 AI 審閱不是單純標差異，而是要輸出「差異 + 法務態度 + 提示語」。

### 4. AI 審閱結果的輸出模式

從附件九可看出，審閱結果目前是逐個差異片段輸出建議，且建議語氣很一致，常見形式有三種：

- 請使用單位自行確認內容正確
- 請確認附件版本或附件編號無誤
- 某些條款原則不建議修改，如需調整建議送法務室審閱

因此，審閱結果其實可標準化為：

- 差異位置
- 差異內容摘要
- 命中的規則
- 建議等級
- 建議文字
- 是否建議升級人工審閱

### 5. 全文審閱與關鍵字審閱的真正差異

兩者最大差異不在 UI，而在判斷引擎深度：

- 關鍵字審閱：看得見差異字串即可
- 全文審閱：要能理解條文上下文、附件語意與整體條款風險

如果要做全文審閱，建議至少補上三個能力：

1. 文件切段與條號辨識
2. 規則命中後的上下文擷取
3. 風險等級與人工覆核條件

## 三、自動化產生合約需求重點

### 1. 使用者需求文件整理

附件一 PDF 提到的核心方向可以整理成兩個模組。

#### 模組一：AI 契約自動生成

目標是讓使用者不用從零手填合約，而是透過兩種方式快速得到初稿：

1. 結構化表單直接輸入
2. 上傳報價單或專案附件，由 AI 自動判讀並預填

之後仍保留人工微調與確認，表示生成不是全自動直出，而是「AI 預填 + 人工確認」。

#### 模組二：聲請單與審核流程整合

文件中描述的流程是：

1. 選擇契約類型
2. 上傳相關附件
3. 填寫表單
4. 提交並成立聲請
5. 進入審核流程

這表示合約生成只是前半段，真正業務價值在於把生成結果接到既有聲請與簽核流程。

### 2. 時程規劃意圖

附件一中的時程說明可讀成：

- 4 月到 5 月先完成模組一
- 先落地 5 種合約類型
- 6 月到 7 月再完成模組二
- 模組二的重點是自動起單與後續簽核

這很明顯是在暗示 POC 應該分兩階段，而不是一次把整條流程做完。

### 3. 既有法務助理審閱功能的升級需求

附件一也提到現有 AI 法務助理審閱合約還要做這些升級：

- 新增契約範本與審閱過契約選項
- 契約範本類型下拉選單支援查詢
- 自動帶入最新版契約範本
- HTML 差異檔改成單一顏色呈現
- 契約類別擴增至全類型契約範本

這代表生成系統與審閱系統應該共享至少三種主資料：

- 契約類型清單
- 最新版範本
- 規則與附件版本

### 4. POC 合約範本結構

附件二是一份帶 placeholder 的業務委任合約範本，明顯適合作為文件生成模板。可辨識的變數包含：

- 專案名稱
- 乙方全名
- 金額
- 負責人
- 地址
- 統編
- 簽約日期

所以 POC 很適合先用「模板 + 欄位映射 + docx 回填」方式做，不需要一開始就做自由生成。

## 四、架構圖解讀

附件三架構圖反映的流程如下：

- 使用者在 Teams 互動
- Teams 對接 Copilot Studio
- Copilot Studio 與 Microsoft Foundry 交互
- Adaptive Card 負責介面
- 主題負責流程編排
- Power Automate 做工具串接
- Python 服務處理文件生成
- 最後輸出 `.docx` 回到 Teams

這張圖透露的不是最終技術細節，而是職責切分：

- Teams：入口與人機互動
- Copilot Studio：對話流程與 agent 編排
- Foundry：模型與推理能力
- Power Automate：流程膠水層
- Python：文件處理執行器

## 五、建議的 POC 切法

### Phase 1：合約生成最小可行版

先做下列能力：

1. 選擇契約類型
2. 以 Adaptive Card 填欄位
3. 支援附件上傳
4. 用 AI 從附件抽出欄位候選值
5. 人工確認欄位
6. Python 將欄位寫入 docx 模板
7. 回傳生成文件

這一階段先不要碰正式簽核，只要做到「可生成、可修改、可下載」。

### Phase 2：審閱與規則共用

把 AI 合約審閱與合約生成接到同一份契約主資料：

1. 範本版本控管
2. 合約類型主檔
3. 規則知識庫
4. 生成後立即啟動自我審閱

這樣可以形成「生成後先做機器審閱」的閉環。

### Phase 3：起單與簽核整合

最後再把生成後的文件與欄位 metadata 串到聲請單與簽核：

1. 自動起單
2. 帶入契約類型與關鍵欄位
3. 附上生成 docx 與來源附件
4. 依規則分流到不同審核節點

## 六、建議的系統邊界

### 1. Python 服務負責

- docx 模板處理
- 差異比對
- 檔案格式轉換
- 欄位抽取後處理

### 2. Foundry / LLM 負責

- 從附件抽取候選欄位
- 對照規則生成審閱意見
- 對差異內容做風險摘要
- 對不完整欄位提出補問

### 3. Copilot Studio / Power Automate 負責

- 對話流程
- 卡片 UI
- 檔案與表單資料搬運
- 呼叫 Python 與 Foundry
- 回傳結果給 Teams

## 八、目前已知風險與待確認事項

### 1. 全文審閱的輸入格式

需先確認 AI 讀的是：

- 原始 docx 轉純文字
- 依條號切段後文本
- 還是含格式與批註的結構化表示

這會直接影響 prompt 與資料前處理。

### 2. 規則版本控管

規則檔現在是 Excel，適合人工維護，但不適合直接進入線上推理。建議後續轉成 JSON 或資料表。

### 3. docx 模板回填方式

POC 若直接依 placeholder 字串替換最省時，但正式版可能要處理：

- 表格欄位
- 重複區塊
- 附件編號連動
- 頁首頁尾

### 4. 生成與審閱的責任界線

目前需求文件很明確地保留人工確認，因此系統定位應是：

- 生成草稿
- 提示風險
- 協助審閱

而不是自動產生最終法務定稿。

4. 製作 Teams Adaptive Card 欄位表單
5. 規劃 Python 文件服務介面，例如 `extract_fields`、`render_docx`、`build_diff_html`、`review_contract`
